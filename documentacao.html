01 — Visão Geral

O Projeto AutoTrack BR

Sistema de histórico veicular inspirado no CarVertical europeu, adaptado ao contexto brasileiro.

O AutoTrack BR é um sistema web completo para registro e consulta de histórico veicular. Qualquer usuário pode pesquisar um veículo pela placa e visualizar eventos como acidentes, roubos, leilões e manutenções, além de ver o preço de mercado atualizado pela Tabela FIPE.

Comparativo com o CarVertical

Histórico de acidentes ✅ Automático (seguradoras) ✅ Registro manual/importado
Registro de roubo/furto ✅ Via INTERPOL / polícia ✅ Registro com status STOLEN
Histórico de leilão ✅ Integrado ✅ Registro com status AUCTION
Preço de mercado ✅ Via Eurotax ✅ Via Tabela FIPE (API oficial)
Fotos do veículo ✅ ✅ Upload e galeria implementados
API de seguradoras ✅ (dados públicos europeus) ⚠️ Não disponível publicamente no Brasil
Nota sobre dados públicos no Brasil No Brasil, o DETRAN e as seguradoras não disponibilizam APIs públicas de consulta. A solução adotada foi integrar a Tabela FIPE (fonte oficial de preços de mercado do governo brasileiro) e permitir o registro manual de eventos com rastreabilidade de fonte.

02 — Frontend

React 19 + Node.js

Single Page Application com autenticação JWT no cliente e comunicação via Axios.

O frontend foi desenvolvido com React 19 utilizando o Create React App, que utiliza Node.js como ambiente de desenvolvimento e build. A aplicação é uma SPA (Single Page Application) — toda a navegação acontece no cliente sem recarregamentos de página.

Configuração do Cliente HTTP (api.js)

Toda comunicação com o backend passa pela instância Axios configurada em api.js, que injeta automaticamente o token JWT em cada requisição e trata erros de autenticação:

import axios from 'axios';

export const BASE_URL = (process.env.REACT_APP_API_URL || 'http://localhost:8080/api/v1')
  .replace(/\/api\/v1$/, '');

const api = axios.create({ baseURL: `${BASE_URL}/api/v1` });

// Interceptor de requisição: injeta token JWT
api.interceptors.request.use(config => {
  const token = localStorage.getItem('token');
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// Interceptor de resposta: trata token expirado (401)
api.interceptors.response.use(
  response => response,
  error => {
    if (error.response?.status === 401) {
      localStorage.removeItem('token');
      window.location.reload();   // redireciona para login
    }
    return Promise.reject(error);
  }
);

Gerenciamento de Estado

O estado da aplicação é gerenciado pelo React Hooks (useState, useEffect) no componente App.js. A navegação entre telas é controlada por um estado tab, sem uso de React Router:

// Tabs principais
const [tab, setTab] = useState('list');     // 'search' | 'register' | 'list'

// Tabs contextuais (abertas ao selecionar um veículo)
const [selectedVehicle, setSelectedVehicle] = useState(null);  // abre tab 'history'
const [editForm, setEditForm] = useState(null);                 // abre tab 'edit'

// Autenticação
const [user, setUser] = useState(null);     // null = não logado
Por que React e não Vue ou Angular? React é a biblioteca frontend mais adotada no mercado brasileiro. O Create React App (Node.js) fornece o ambiente completo: servidor de desenvolvimento, bundler Webpack, transpilador Babel e suporte a TypeScript. O requisito "front em Node.js" é satisfeito, pois toda a toolchain React roda em Node.js.

Configuração de Segurança (SecurityConfig.java)

A segurança é configurada como stateless — nenhuma sessão é mantida no servidor. Cada requisição deve apresentar um JWT válido no header Authorization:

@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
      .csrf(csrf -> csrf.disable())
      .cors(Customizer.withDefaults())
      .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
      .authorizeHttpRequests(auth -> auth
          .requestMatchers(HttpMethod.OPTIONS, "/**").permitAll()
          .requestMatchers("/api/v1/auth/**").permitAll()           // login/registro público
          .requestMatchers(HttpMethod.GET, "/api/v1/photos/**").permitAll()
          .requestMatchers("/swagger-ui.html", "/swagger-ui/**").permitAll()
          .anyRequest().authenticated()                             // todo o resto exige JWT
      )
      .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);
    return http.build();
}

04 — Banco de Dados

05 — Repositórios

Spring Data JPA

Interfaces de acesso ao banco — sem SQL manual, geração automática de queries.

O Spring Data JPA elimina a necessidade de escrever código JDBC ou SQL na maioria dos casos. Basta declarar uma interface que estende JpaRepository e o Spring gera a implementação automaticamente em tempo de execução.

VehicleRepository

@Repository
public interface VehicleRepository extends JpaRepository<Vehicle, Long> {

    // Spring gera: SELECT * FROM vehicles WHERE plate = ?
    Optional<Vehicle> findByPlate(String plate);

    // Spring gera: SELECT * FROM vehicles WHERE vin = ?
    Optional<Vehicle> findByVin(String vin);

    // Spring gera: SELECT COUNT(*) > 0 FROM vehicles WHERE plate = ?
    boolean existsByPlate(String plate);
}

VehicleHistoryRepository

@Repository
public interface VehicleHistoryRepository extends JpaRepository<VehicleHistory, Long> {

    // Busca todos os eventos de um veículo, ordenados por data decrescente
    List<VehicleHistory> findByVehicleIdOrderByEventDateDesc(Long vehicleId);
}

Padrão Repository O repositório é a camada que isola o acesso ao banco de dados. Os Services nunca falam diretamente com o banco — eles usam os repositórios. Isso facilita testes (pode-se trocar o banco por um mock) e mantém o código organizado.

06 — Camada de Serviços

Business Logic Layer

Regras de negócio separadas dos controllers e dos repositórios.

Os Services são responsáveis pelas regras de negócio da aplicação. Eles recebem chamadas dos Controllers, aplicam validações e lógica, e usam os Repositories para persistir ou consultar dados.

VehicleService — Regras de Negócio

@Service
@RequiredArgsConstructor
public class VehicleService {

    private final VehicleRepository vehicleRepository;

    public VehicleResponseDTO create(VehicleRequestDTO dto) {
        // REGRA: placa não pode ser duplicada
        if (vehicleRepository.existsByPlate(dto.getPlate().toUpperCase())) {
            throw new BusinessException("Placa já cadastrada: " + dto.getPlate());
        }

        Vehicle vehicle = Vehicle.builder()
            .plate(dto.getPlate().toUpperCase())    // REGRA: placa sempre em maiúsculas
            .brand(dto.getBrand())
            .model(dto.getModel())
            .year(dto.getYear())
            .status(dto.getStatus() != null ? dto.getStatus() : VehicleStatus.REGULAR)
            .fipeCode(dto.getFipeCode())
            .fipePrice(dto.getFipePrice())
            .fipeReference(dto.getFipeReference())
            .build();

        return toDTO(vehicleRepository.save(vehicle));
    }
}

FipeService — Integração com API Externa

O FipeService atua como proxy entre o frontend e a API da FIPE, resolvendo o problema de CORS (a API da FIPE não permite chamadas diretas do browser). Inclui cache de 12 horas para o código de referência mensal:

@Service
public class FipeService {

    private Integer cachedRefCode;       // cache do código de referência mensal
    private LocalDateTime cacheTime;

    private Integer getRefCode() {
        // Reutiliza o código por 12 horas — ele muda apenas mensalmente
        if (cachedRefCode == null || cacheTime.plusHours(12).isBefore(LocalDateTime.now())) {
            // busca o código atual na API FIPE
            cachedRefCode = buscarCodigoAtual();
            cacheTime = LocalDateTime.now();
        }
        return cachedRefCode;
    }

    public List<Map<String, Object>> getMarcas(int tipo) {
        MultiValueMap<String, String> body = new LinkedMultiValueMap<>();
        body.add("codigoTabelaReferencia", getRefCode().toString());
        body.add("codigoTipoVeiculo", String.valueOf(tipo));
        return post("/ConsultarMarcas", body, List.class);
    }
}

Tratamento Global de Erros

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<Map<String, String>> handleBusiness(BusinessException ex) {
        return ResponseEntity.badRequest()
            .body(Map.of("message", ex.getMessage()));
    }

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<Map<String, String>> handleNotFound(ResourceNotFoundException ex) {
        return ResponseEntity.status(404)
            .body(Map.of("message", ex.getMessage()));
    }
}

09 — Maturidade RESTful

Richardson Maturity Model

Modelo que classifica APIs REST em 4 níveis de maturidade.

O Richardson Maturity Model (criado por Leonard Richardson) define 4 níveis para classificar o quão "RESTful" uma API é. O AutoTrack BR foi implementado no Nível 2, que é o padrão adotado pela indústria.

0
The Swamp of POX
Um único endpoint, tudo via POST. Sem recursos, sem verbos.
1
Resources
URLs representam recursos distintos. Ex: /vehicles, /users.
2
HTTP Verbs ← AutoTrack
GET, POST, PUT, DELETE com status codes corretos. Padrão da indústria.
3
HATEOAS
Responses incluem links de navegação. Raramente usado na prática.

Como o AutoTrack atende o Nível 2

✅ Recursos bem definidos (Level 1):

GET  /api/v1/vehicles          ← coleção de recursos
GET  /api/v1/vehicles/{id}     ← recurso específico
GET  /api/v1/vehicles/{id}/history  ← sub-recurso

✅ Verbos HTTP com semântica correta (Level 2):

VerboOperaçãoStatus Code retornadoIdempotente?
GETConsultar / listar200 OK✅ Sim
POSTCriar novo recurso201 Created❌ Não
PUTAtualizar recurso existente200 OK✅ Sim
DELETERemover recurso204 No Content✅ Sim

✅ Status Codes semânticos:

200 OK           → consulta bem-sucedida
201 Created      → recurso criado com sucesso (POST)
204 No Content   → exclusão bem-sucedida (DELETE)
400 Bad Request  → dados inválidos ou regra de negócio violada
401 Unauthorized → JWT ausente ou expirado
404 Not Found    → veículo/evento não encontrado
Por que Level 2 e não Level 3 (HATEOAS)? O Level 3 exige que cada response inclua links para ações relacionadas (ex: ao retornar um veículo, incluir link para seu histórico). Na prática, isso é implementado por menos de 5% das APIs comerciais — incluindo Netflix, Stripe, GitHub e PayPal, que param no Level 2. O Level 2 é o padrão consolidado do mercado.

10 — Integração Externa

Tabela FIPE

Proxy Spring Boot para a API oficial de preços de veículos do governo brasileiro.

A Tabela FIPE (Fundação Instituto de Pesquisas Econômicas) é a referência oficial de preços de veículos no Brasil, usada por seguradoras, bancos e concessionárias. O AutoTrack integra essa tabela para fornecer o valor de mercado atualizado de cada veículo.

Por que um Proxy no Backend?

A API da FIPE bloqueia requisições diretas de navegadores por segurança (política CORS). A solução é criar um proxy no Spring Boot que:

  1. Recebe a requisição do React
  2. Repassa para a API da FIPE com os headers corretos (Referer)
  3. Retorna a resposta ao frontend

Implementação do Proxy (FipeService.java)

@Service
public class FipeService {

    private static final String FIPE_BASE = "https://veiculos.fipe.org.br/api/veiculos";
    private final RestTemplate restTemplate = new RestTemplate();

    // Cache de 12h do código de referência (muda mensalmente)
    private Integer cachedRefCode;
    private LocalDateTime cacheTime;

    private HttpHeaders buildHeaders() {
        HttpHeaders h = new HttpHeaders();
        h.setContentType(MediaType.APPLICATION_FORM_URLENCODED);
        h.set("Referer", "https://veiculos.fipe.org.br/");  // header obrigatório
        return h;
    }

    public List<Map<String, Object>> getMarcas(int tipo) {
        MultiValueMap<String, String> body = new LinkedMultiValueMap<>();
        body.add("codigoTabelaReferencia", getRefCode().toString());
        body.add("codigoTipoVeiculo", String.valueOf(tipo));  // 1=carro, 2=moto, 3=caminhão
        return post("/ConsultarMarcas", body, List.class);
    }

    public Map<String, Object> getPreco(int tipo, int marca, int modelo, String ano) {
        MultiValueMap<String, String> body = new LinkedMultiValueMap<>();
        body.add("codigoTabelaReferencia", getRefCode().toString());
        body.add("codigoTipoVeiculo",      String.valueOf(tipo));
        body.add("codigoMarca",            String.valueOf(marca));
        body.add("codigoModelo",           String.valueOf(modelo));
        body.add("ano",                    ano);
        body.add("tipoPesquisa",           "2");
        return post("/ConsultarValorComTodosParametros", body, Map.class);
    }
}

Fluxo de Consulta no Frontend (FipeSearch.js)

A interface guia o usuário em 4 seleções em cascata até obter o preço:

1. Selecionar Tipo   → GET /api/v1/fipe/marcas?tipo=1

3. Selecionar Modelo → GET /api/v1/fipe/anos?tipo=1&marca=22&modelo=5
4. Selecionar Ano    → GET /api/v1/fipe/preco?tipo=1&marca=22&modelo=5&ano=2020-1
